1. 总体结论

FreeCode 的 session 管理是“事件溯源式”的设计：运行时只维护少量全局会话状态，真正的会话内容以 JSONL transcript 追加写入。恢复时再从 transcript 解析消息、元数据和快照，重建 UI、模型上下文、文件历史、成本、工作树、agent 设置等状态。

核心设计思想：session 文件是事实来源；内存状态是当前进程的缓存；恢复和分支都围绕 transcript 重建，而不是依赖一个复杂的数据库或长期驻留的会话对象。
创建启动时在 bootstrap/state.ts 生成随机 sessionId。
记录REPL/SDK 每轮把消息和元数据追加到 .jsonl。
检索/resume 扫描项目目录或所有项目目录，构造 LogOption。
恢复切换 sessionId，重建消息链、状态快照、工作树和 agent 上下文。
扩展远端 ingress、bridge、subagent、hooks、compact 都挂到同一日志模型上。

2. 能力模块地图

模块职责关键设计点
src/bootstrap/state.ts 全局 session identity 和进程级状态。 sessionId、parentSessionId、sessionProjectDir 原子切换；区分 originalCwd 和 projectRoot。
src/utils/sessionStorage.ts transcript 路径、追加写入、读取解析、metadata 管理、日志列表、远端 hydration。 内部 Project 类维护当前 session 文件、pending entries、写队列、metadata cache。
src/types/logs.ts 定义 JSONL entry schema。 TranscriptMessage 使用 parentUuid 构成链；其他 entry 记录 title、tag、worktree、PR、快照、内容替换。
src/QueryEngine.ts SDK/headless 查询生命周期。 在模型响应前先持久化用户消息，降低进程被杀导致不可恢复的风险。
src/screens/REPL.tsx 交互式 UI 会话主循环。 持有 resume 回调，恢复 AppState、成本、agent、worktree、remote agent tasks。
src/commands/resume /resume 命令和选择器。 支持 picker、UUID、custom title、同 repo worktree、跨项目提示命令。
src/commands/branch 当前会话分支。 复制主链消息，改写 sessionId，写入 forkedFrom 和 content replacement。
src/services/api/sessionIngress.ts 远端 session 日志追加/读取。 按 session 顺序写；使用 Last-Uuid 乐观并发控制；409 时可采用服务端 head 重试。

3. 数据模型：JSONL Transcript 作为事实来源

session 文件位于配置目录的项目子目录中，路径由 getTranscriptPath() 计算：{claudeConfigHome}/projects/{sanitizedProject}/{sessionId}.jsonl。如果当前 session 来自其他项目或 worktree，sessionProjectDir 会覆盖默认项目目录。

3.1 核心 entry 类型

类型作用恢复时如何使用
TranscriptMessage 用户、助手、附件、系统消息。包含 sessionId、uuid、parentUuid、cwd、timestamp、gitBranch。 通过 parentUuid 从 leaf 回溯构造对话链。
custom-title、ai-title、tag 会话展示和搜索元数据。 /resume 列表、标题匹配、退出提示使用。
file-history-snapshot 文件历史快照。 恢复 file history，用于 rewind、diff、上下文还原。
attribution-snapshot 提交归因状态。 恢复 Claude 贡献统计。
worktree-state 记录会话最后是否在 worktree 中。 resume 时尝试 cd 回 worktree，路径不存在则写回 null。
content-replacement 记录大工具结果被替换成 stub 的决策。 恢复 prompt cache 稳定性，避免重新发送完整内容。
marble-origami-commit/snapshot 上下文折叠状态。 恢复 context collapse 的提交日志和 staged queue。

3.2 链式会话模型

TranscriptMessage {
  uuid: UUID
  sessionId: UUID
  parentUuid: UUID | null
  isSidechain: boolean
  type: "user" | "assistant" | "attachment" | "system"
}

这不是简单数组恢复，而是图链恢复。loadTranscriptFile() 先把所有 transcript message 放入 map，再计算 leaf，最后 buildConversationChain() 从最新 leaf 沿 parentUuid 回溯。这个设计天然支持分支、fork、compact 后的链截断，以及 sidechain/subagent transcript。

4. 写入链路：追加日志、去重和顺序保证

核心写入函数是 recordTranscript(messages)。它先加载当前 session 已写入的 message UUID 集合，过滤已记录消息，再把新增消息交给 Project.insertMessageChain() 追加到 JSONL。

去重策略

通过 getSessionMessages(sessionId) 缓存已写 UUID。对于 growing-array 调用方，已写消息作为前缀会更新 parent cursor；对于 compaction 保留段，已写消息出现在新增 compact boundary 后，不会污染新链。

写队列策略

Project 维护 per-file queue，默认 100ms drain；大批量按 100MB chunk 分段 append。flushSessionStorage() 可等待所有 pending writes。

4.1 SDK/Headless 写入

QueryEngine.submitMessage() 会在进入模型 query loop 前先持久化用户消息。这样即使 API 响应前进程被杀，session 仍然至少能恢复到“用户输入已接受”的状态。

processUserInput()
  -> mutableMessages.push(user messages)
  -> recordTranscript(messages)   // 先写用户输入
  -> query(...)
  -> assistant/user/compact boundary yielded
  -> recordTranscript(messages)
  -> flushSessionStorage() on eager/cowork/result

4.2 交互式 REPL 写入

REPL 路径主要通过 useLogMessages 观察 UI message array 增量写入，同时在 resume、clear、compact 等路径主动调用 sessionStorage 函数，确保链路和 metadata 对齐。

5. 恢复链路：从 LogOption 重建运行态

/resume 命令负责选择目标会话；实际恢复由 REPL 的 resume(sessionId, log, entrypoint) 或启动路径的 processResumedConversation() 完成。

1. 找会话loadSameRepoMessageLogs() 或 loadAllProjectsMessageLogs() 扫描 JSONL。
2. 载入完整日志lite log 经 loadFullLog() 转成完整 LogOption。
3. 反序列化deserializeMessages() 清理未配对 tool use 等异常。
4. 切换身份switchSession(sessionId, dirname(fullPath)) 原子切 sessionId 和项目目录。
5. 恢复状态file history、cost、agent、worktree、remote tasks、metadata。

5.1 关键恢复动作

  • restoreSessionStateFromLog()：恢复 file history、attribution、context collapse、TodoWrite 状态。
  • restoreAgentFromSession()：恢复自定义 agent 和 agent model override。
  • restoreWorktreeForResume()：如果 transcript 记录了 worktree，尝试切回对应目录，并清理 CLAUDE.md/system prompt/plans 缓存。
  • resetSessionFilePointer() + adoptResumedSessionFile()：让当前进程后续写回原 session 文件，而不是误写新文件。
  • restoreSessionMetadata()：恢复 title、tag、agent name/color、mode、PR link 等尾部元数据缓存。
跨项目恢复约束：/resume 对不同项目的会话不会直接恢复，而是提示用户在对应目录运行命令；同一 repo 的 worktree 可以直接恢复。这避免 session 的文件路径、权限和上下文绑定错位。

6. 分支能力：复制 transcript 形成新 session

/branch 通过读取当前 session JSONL、筛选主会话消息、改写 sessionId 后写入一个新的 session 文件。分支不是数据库层的 copy-on-write，而是 transcript 层的显式复制。

6.1 分支步骤

  1. 生成新的 forkSessionId。
  2. 读取当前 getTranscriptPath()。
  3. 过滤 isTranscriptMessage(entry) && !entry.isSidechain。
  4. 为每条消息改写 sessionId、重建 parentUuid、写入 forkedFrom 溯源字段。
  5. 复制主线程 content-replacement 记录，避免恢复后大工具结果重新膨胀。
  6. 保存唯一标题，如 原标题 (Branch)、原标题 (Branch 2)。
  7. 调用 REPL resume 回调进入新分支。

这个实现强调可恢复性和 prompt cache 稳定性。它保留原消息 UUID 作为 forkedFrom.messageUuid 的溯源，同时在新链里用新的 sessionId 让恢复和展示归属清晰。

7. 压缩、大文件与恢复性能

session 文件可能增长到很大，代码里有多处专门为长会话设计的优化和故障修复。

问题实现策略价值
JSONL 过大导致 OOM MAX_TRANSCRIPT_READ_BYTES、SKIP_PRECOMPACT_THRESHOLD、readTranscriptForLoad()、walkChainBeforeParse()。 只读取 compact boundary 之后的有效段，减少内存峰值。
compact 前 metadata 被跳过 scanPreBoundaryMetadata() 额外扫描 boundary 前 metadata。 恢复 title、tag、agent、mode、PR link 等元数据。
历史 progress message 破坏 parent chain progressBridge 把 legacy progress parent 桥接到非 progress 祖先。 兼容旧 transcript，不让恢复链断裂。
preserved segment 需要重连 applyPreservedSegmentRelinks() 在解析后修正 parent 关系。 保证 compact 后保留段仍在恢复上下文里。
metadata 只在文件尾部快速读取 reAppendSessionMetadata() 在 compaction 和 exit 时重写尾部 metadata。 提高列表读取、resume picker、退出提示的可靠性。

8. 远端会话、Bridge 与 Hooks

8.1 Session Ingress

services/api/sessionIngress.ts 提供远端日志 append 和 hydration。写入使用 JWT token，按 session 顺序执行，并通过 Last-Uuid header 做乐观并发控制。遇到 409 时，会读取服务端 last uuid 或重新拉取日志，再重试当前 entry。

appendSessionLog(sessionId, entry, url)
  -> getOrCreateSequentialAppend(sessionId)
  -> PUT entry with Last-Uuid
  -> 200/201: update lastUuidMap
  -> 409: adopt server head and retry
  -> 401: fail fast

8.2 Hydration

hydrateRemoteSession(sessionId, ingressUrl) 会先 switchSession()，再从远端拉取 entry，覆盖写入本地 JSONL。CCR v2 路径还支持从内部事件读取 foreground transcript 和 subagent transcript。

8.3 Hooks

session 生命周期还接入 SessionStart 和 SessionEnd hooks。REPL resume 前会先执行当前 session 的 SessionEnd，再对新 session 执行 SessionStart，让外部自动化看到明确的 session 边界。

9. 设计评价与技术细节总结

优势
  • JSONL 追加模型简单、可检查、崩溃后容易恢复。
  • parentUuid 链支持分支、compact 截断、sidechain 和 subagent。
  • sessionId 和 sessionProjectDir 原子切换，减少跨 worktree/跨项目错写。
  • metadata tail re-append 兼顾快速列表和长期 session 可靠性。
  • SDK 在模型响应前先写用户输入，对异常退出更友好。
复杂度与风险
  • sessionStorage 承担职责过多：路径、写队列、解析、metadata、remote hydration 都在同一文件。
  • 恢复正确性高度依赖 parentUuid、compact boundary、content replacement 的细节约定。
  • 多入口写入场景复杂：REPL、SDK、remote、subagent、branch 都可能写 transcript。
  • 大文件优化依赖启发式扫描和 kill switch，测试覆盖必须跟上。

9.1 能力边界

能力当前设计成熟度说明
本地会话持久化高完整 JSONL append、flush、dedup、metadata 缓存。
会话恢复高支持 picker、UUID、标题、worktree、agent、cost、file history。
会话分支中高主链复制完善，但不是增量引用，长会话会复制大量日志。
大文件恢复高但复杂有针对 OOM 和 compact 的多层优化，维护门槛较高。
远端同步中高有顺序写和冲突恢复；仍依赖 token、网络、服务端协议。
统一抽象中架构上更像 transcript plumbing，而不是清晰分层的 session domain service。

9.2 建议的架构演进方向

  • 把 sessionStorage.ts 拆成 transcript writer、transcript reader、metadata service、remote sync 四个边界，降低维护成本。
  • 为 Entry 做 schema version 和迁移层，减少旧 transcript 兼容逻辑散落在 loader 中。
  • 把 resume 恢复步骤封装为显式 pipeline，减少 REPL 和 startup path 的重复。
  • 对 branch 引入可选轻量引用或按 leaf 截断复制，降低长会话分支成本。
  • 为 compact/resume 构造更多 fixture：legacy progress、preserved segment、content replacement、cross-worktree、remote hydration。
